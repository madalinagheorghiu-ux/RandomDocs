Versioning tree — branch & version UX spec

CORE-4974 · merge any branch into any branch + continue after merge · v0.5 · 2026-07-16 (milestone version-E) · owner: Madalina

Consolidates the UX rules validated in the standalone prototype (/Users/madalina/flowx-code/versioning-tree-prototype/, run with python3 -m http.server 8899 --bind 127.0.0.1). The prototype renders with the real @flowx/gitgraph engine (vendored GitgraphJS fork from designer/libs/gitgraph) and the production ApplicationVersion[] history shape, so every rule here maps 1:1 onto the product model: versions form linear chains per branch via previousVersionId; merges reference mergedFromVersionId; one WIP per branch, always at the tip.

1 · Reading the tree

Node states

NodeMeaning
Filled dot (branch color)Committed version
Dashed hollow dotWork-in-progress (WIP) version
Hatched rounded squareMerge in progress — a merge landed here and its conflicts must be resolved before it completes
Outer solid ring (branch accent)Latest version on the branch you're working on ("you are here")

Legend (order + neutral swatches): items read selected · work in progress · committed · merge in progress · imported · merge, drawn in neutral grays (#64748b / #e3e8ed / #a6b0be) — deliberately not branch colors, since the legend explains node shapes/states, not any one branch's hue. The selected swatch is a border-only ring (no fill); the old "latest" filled-dot entry was dropped from the legend (the current-version marker still renders in the graph).

Making main unmistakable

Branch labels are positioned by the engine at each branch tip, so the main tag can appear between feature tags. Three stacked cues compensate:

  • Filled blue chip label (white bold text) — feature branches get outlined pastel chips
  • Thicker line (4px vs 3px)
  • Navigator: main pinned first, blue left accent, blue name, ● default pill

Branch colors

main is always blue; feature branches cycle yellow → green → orange → red (FlowX palettes: line = 200-step, node = 500-step, label bg = 50-step). Assignment is pinned to the project's full branch order so a branch keeps its color when siblings are archived or filtered.

2 · Branch states, ownership, filters

StateDefinitionPill
defaultmain — cannot be archived; every branch eventually flows back here● default
activeTip version is not merged anywhere — the branch has unmerged changes● active
mergedTip version has been merged somewhere (branch is "closed")● merged
archivedSoft-deleted; hidden from the tree entirely, listed in the Archived section with its archive date; all versions remain and can be brought back● archived
Requirement 2 — a merged branch stays editable (merged is a tip-state, not history). The mergedInAnyBranch freeze never blocks further work: the user can start a new version on a merged branch and merge it again (re-merge). A branch that was merged and then received new commits ("continued after merge") is active again; the list annotates it "· continued after merge". Merge history never permanently closes a branch.

Ownership & filters

  • Every branch shows its owner; the viewer's own branches get a you chip ("MyBranches" = branches created by me).
  • Filters (segmented control), in this order: All / My branches / Active / Merged.
  • Filters dim non-matching branches instead of hiding them — topology stays readable and merge edges never dangle.
  • Archived branches are hidden from the graph and the branch list; their home is the Archived section (§6).

3 · Selection & branch evolution

  • Default view on load / scenario switch: focus the current (latest) committed version you're working on — details card leads the panel. A merge-in-progress version is never seeded; that state appears only transiently while a merge is resolved (§5). Passive: graph keeps the plain "All" look; fading/ring begin on the first real click.
  • Click any node, branch label, or list row → selects that branch (clicking a node/timeline entry targets that version). Right-click a bubble → selects it and opens a context menu with that version's full action set. Selection persists across filter changes (the details card no longer clears when a segment is clicked). There is no ✕ / clear-selection anymore — the panel always leads with a selected version; switch by clicking another node/branch.
  • Selection fades the rest — lighter, never gray. The selected branch keeps its exact normal appearance (no restyling); every other branch drops to ~28% opacity, becoming a lighter version of its own color, never a gray substitute. The fade is applied in place on the rendered graph (no re-render), so selecting a node never flickers. Filters de-emphasize non-matching branches the same way.
  • The selected bubble is ringed in the branch's deep tone — like the current-version marker in the legend, but darker so "selected" reads distinct from "current". Applied as a DOM mutation on the live graph (no rebuild); clicking a different node or timeline entry moves the ring.
  • Panel layout — details lead, navigator is light. The branch list is a compact single-line navigator (swatch · name · you · state dot). When a branch is selected its Details card leads the panel. The details header shows the branch name plus two icon buttons (replacing the old ✕): an edit-name pencil (rename the branch — every version moves to the new name, color preserved; disabled on main/imported/archived) and a download icon (export the selected version).
  • Details panel: state explanation in plain language, owner, created-from, merges in, merged into, then an Evolution timeline and the actions (§4).
  • Evolution timeline row: committer avatar (initials, full name on hover), commit message truncated with … (full on hover), and the date underneath. On the selected committed row an inline edit-message pencil appears — the only place to rename a version's message; it follows the selection. Badges wrap below: ⭠ merge from X ⭢ merged into Y build-n latest work in progress
  • Archived interactions stay legible: wherever a version interacted with a now-archived branch (merge from/into, created-from), the badge or name carries a dashed archived tag. Clicking it opens the archive-details modal (when / by whom / reason + Unarchive) — see §6.
  • Selecting a branch from the Archived section shows its details and Evolution timeline (panel only — archived branches don't render in the graph), with an archived pill next to the title.

4 · Actions — Version actions vs Branch actions

The panel groups actions under two headers: Version actions operate on the selected version; Branch actions operate on the branch as a whole.

Version actions — Committed 4-button grid: Create build (disabled once created) · Create branch · Start version (new WIP at tip; disabled if the tip is already in progress) · Delete version (danger — tip only & not merged, else disabled with reason). 5 · Merge rules (any-to-any)

Requirement 1 — merge any branch into any branch. Target candidates are all other branches, never "ancestors only": siblings into siblings, descendants into unrelated branches, and main into a feature branch are all first-class merges. Eligibility is state-based only (WIP / fully-merged / archived) — ancestry is never a criterion. Blocked targets stay visible in the target list, disabled with their reason, so the rule teaches itself.

Source: the committed tip of a branch (product canMerge: status === COMMITTED && lastVersionInBranch). A WIP can never be a merge source; any branch — including main and already-merged branches (re-merge) — can be one.

Targets:

TargetAllowedReason / copy
main (tip committed)✓Always the default destination
Active branch, tip committed✓Includes merged-then-continued branches
Branch with WIP at tip✕"Commit or delete the WIP changes first"
Fully merged branch (tip merged, no new work)✕"Start a new version to reopen this branch first"
Archived branch✕Excluded from target list
Why WIP targets are blocked (validated against product code — merge is disabled for WIP-tipped targets in version-actions.component.ts): merging into the WIP would mix the merge with unreviewed, uncommitted edits (git's dirty-tree problem); merging underneath the WIP onto the last committed version would fork the linear chain — the WIP would not receive the merge, and committing it later would silently drop the merged changes from the branch tip. Blocking forces the WIP owner to decide.
Why fully-merged targets are blocked: keeps "merged = done" trustworthy; reopening is an explicit act (Start new version). This is a product-policy choice, not a technical necessity — git would allow it.
  • Two-step merge, chained (conflicts): confirming ("Start merge") lands a merge-in-progress version on the target (status MERGE_IN_PROGRESS, the hatched-square bubble) and immediately opens the Resolve-conflicts modal. Resolve via Resolve conflicts (→ committed merge, source marked merged), or close it via Resolve later / Esc / ✕ — which keeps the merge-in-progress version parked in the graph (status preserved), not removed (changed 2026-07-16; earlier it discarded). A parked merge is resumed from its Version actions: Resolve conflicts (completes it) or Discard merge (the only way, besides completing, to remove it). The source branch is untouched; a branch whose tip is merge-in-progress is "busy" (can't merge, start a version, or be archived) until resolved. The resolve modal is a placeholder for the product's merge-conflicts editor.
  • Re-merge is allowed: a branch can merge into the same target repeatedly (scenario c) and in both directions (scenario e).
  • Merge modal: pick target from the valid list; blocked targets appear disabled with their reason, so the rule teaches itself.
Future option, out of v1: autostash-style "move my WIP on top of the merge" (stash WIP → merge onto committed tip → replay WIP on the merge result). Answers "the WIP gets the merge anyway", but can trigger a second conflict pass inside the WIP. Test whether users ask for it.

6 · Archive (soft delete)

There is no hard delete. "Delete branch" is an archive: the branch is hidden from active work, every version remains in the tree, and it can be unarchived at any time (plus an immediate Undo pill).

Eligibility — any branch except main, and not mid-merge

main is blocked ("The default branch cannot be archived."), and a branch whose tip is a merge-in-progress is blocked too ("Resolve or discard the in-progress merge before archiving this branch."). Everything else can be archived. The confirm modal captures an optional reason and records who archived it and when, and spells out the impact so archiving an unmerged or in-progress branch is a conscious choice:

Branch conditionImpact message in the confirm modal
Unmerged changes (state ≠ merged)⚠ "It has unmerged changes — they are not part of any other branch and stay parked until you unarchive and merge."
Has a WIP version⚠ "Its work-in-progress version is parked untouched — nothing is committed or discarded."
Holds the current version⚠ "You are working on this branch right now — the current version is hidden until you unarchive."
Was merged into other branches"Versions already merged into X are kept there." (reassurance, not a warning)

Presentation & recovery

  • Archive is a small danger-secondary button (white fill, red text, neutral border — styled like Delete version) in the "Branch actions" footer.
  • Archived branches are hidden from the graph and the branch navigator; they live in a collapsed disclosure ("▸ Archived · N") at the bottom of the navigator, expanded on demand. Each archived row is compact — swatch, name, archive date, and an inline Unarchive button.
  • Excluded from merge targets while archived.
  • Unarchiving always goes through the archive-details modal. Both the dashed archived tag and the Archived-disclosure Unarchive CTA open the same modal — when / by whom / reason — with Cancel and Unarchive branch. On unarchive the branch returns to the tree and is focused (selected in the graph + its details card shown).
  • Two ways in, plus quick revert: the archive-details modal (from a tag or the Archived CTA), or the Undo pill right after archiving.

7 · Imported (read-only) branches & versions

How an imported application version — brought in with its own branches and versions — reads in the tree. View-only is a per-branch property (driven by imported: true), so imported branches are read-only wherever they appear while the real branches around them stay fully editable.

  • Woven into the Demo / full story (primary): imported-core (2 versions) + its sub-branch imported-reporting sit grayed inside the main tree, alongside the editable branches. Selecting an imported branch shows info + Evolution but no actions; a real branch keeps all actions. Imported branches are excluded from merge targets (can't merge into a read-only branch). This is the integration the product should mirror.
  • Standalone imported scenario ("Imported · view-only"): a whole-scenario viewOnly variant (main + feature-live + imported branches) for a focused, actions-free view — redundant now that the demo integrates imported.
  • Rendering: imported versions are grayed (neutral palette) and each imported node draws the imported glyph (gray circle + down-arrow ↓, matching the legend); labels, navigator swatches, and the details imported pill are gray; branch state is imported.
  • Mechanism: per-version imported: true drives gray palette + glyph + imported state, suppresses that branch's actions in any scenario, and drops it from merge-target lists. SCENARIOS.imported.viewOnly === true additionally makes the standalone scenario read-only for every branch.
Open (for testing): final imported-node glyph (currently ↓-in-circle); whether imported nodes should reflect WIP/merge states; and where imported branches sit in the demo (currently branch off the initial version → bottom lanes).

8 · Open questions for user testing

  • Fully-merged branch as merge target: keep the two-step "reopen, then merge", or auto-reopen? (§5 policy choice.)
  • Archiving a branch that holds the user's current version is allowed with a warning — should the product instead move the current-version pointer somewhere visible (e.g., main's tip)?
  • Autostash-style WIP handling on merge targets (§5).
  • Merge-line color follows the target branch — is that legible enough, or do merge edges need their own affordance (e.g., source-colored edge)?
