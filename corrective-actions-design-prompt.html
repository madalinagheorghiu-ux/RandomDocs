Claude Design Prompt

Corrective Actions
Runtime Operations Prototype

Design a powerful, safe operator console for intervening on running process instances in real time — inside a BPMN process orchestration platform (FlowX).

Context & Mental Model

In FlowX, a Process Instance is a running execution of a BPMN process definition. Each instance carries one or more Tokens — markers that travel through the process graph node by node. A token's position tells you exactly where execution is stuck or progressing.

Process instances can be launched standalone or as part of a UI Flow Session — a higher-level orchestration that groups multiple process instances into a single end-user journey (e.g. an onboarding flow that spawns a KYC process, a document upload process, and an approval process together).

An application can have multiple builds — versioned snapshots of the process definitions and configuration deployed at a specific point in time. A running instance always belongs to a specific build. When a new build is deployed (e.g. after a hotfix), existing running instances remain on their original build unless explicitly migrated.

Operators need to intervene when: a token is stuck at a broken node; running instances need to continue on a newer build after a hotfix; or a variable value needs correction without restarting the instance from scratch.

Data Model

Process Instance

  • uuid — unique identifier
  • processDefinitionName — process name
  • state — see statuses below
  • buildAppVersionId — current build
  • initiatorUsername — who started it
  • timestamp — when it started
  • tokens[] — list of tokens
STARTED ON_HOLD FINISHED FAILED EXPIRED ABORTED TERMINATED

Token

  • uuid — unique identifier
  • state — ACTIVE, ON_HOLD, INACTIVE, EXPIRED, ABORTED, DISMISSED, TERMINATED
  • currentNode — node where token sits
  • nodeType — USER_TASK, SERVICE_TASK, EXCLUSIVE_GATEWAY, PARALLEL_GATEWAY, etc.

Build

  • id — unique identifier
  • versionLabel — e.g. v2.1.4
  • deployedAt — deployment timestamp
  • status — ACTIVE, INACTIVE

UI Flow Session

  • id — unique identifier
  • uiFlowName — name of the UI flow
  • status — mirrors process states
  • dateStarted — when it started
  • processInstances[] — grouped instances

Features to Design

1

Move Token New

Moves a token from its current node to any other node in the same process definition. Used when a token is stuck at a broken node and needs to be rerouted to unblock execution.
Design Requirements
  • Works on individual instances (from instance detail view or from list)
  • Works in bulk — operator selects multiple instances of the same process definition and moves all their tokens to the same target node in one action
  • Target node is selected from a visual representation of the BPMN process — show a simplified node list or mini-graph with node names and types
  • Available on instances in any non-terminal state (STARTED, ON_HOLD, CREATED, FAILED, EXPIRED)
  • Can be combined with Change Variables and/or Migrate Instance in a single atomic operation (see Feature 4)
  • Confirmation step: "You are about to move [N] token(s) from node [X] to node [Y] across [M] instance(s). This action cannot be undone."
2

Migrate Running Instance to Another Build New

Moves a running process instance from the build it is currently executing on to a different (typically newer) build of the same application. The instance continues execution from where it left off, but now runs against the updated process definition. Used after deploying a hotfix so instances on the old build continue on the new one without being restarted or lost.
Design Requirements
  • Works on individual instances and in bulk (select multiple → migrate all to target build)
  • Operator selects the target build from a dropdown showing available builds (version label + deployed date + status)
  • The current build each instance is on must be clearly visible before and after selection
  • Bulk migration: all selected instances must belong to the same process definition; warn clearly if mixed selection is attempted
  • Applies to instances in any state — including FAILED, EXPIRED (not just active ones)
  • Can be combined with Move Token and/or Change Variables in a single atomic operation (see Feature 4)
  • Confirmation step: current build → target build, number of instances affected, warning if target build has structural changes that may affect token positions
3

Change Variables

Updates the process variables (key-value pairs) on a running instance. Used to fix bad input data, override a stuck condition variable, or inject missing values without restarting the instance.
Design Requirements
  • Works on instances in any status — including FAILED, TERMINATED, EXPIRED (not just STARTED/ON_HOLD)
  • Single instance: JSON editor with diff view showing current values vs. proposed changes side by side
  • Bulk: apply the same variable patch (partial set of key-value pairs — only the keys being changed) across multiple selected instances simultaneously
  • Bulk editor must make it visually explicit this is a partial patch, not a full replace
  • Can be combined with Move Token and/or Migrate Instance in a single atomic operation (see Feature 4)
4

Combined Corrective Action New

Compose a single corrective operation that applies multiple corrections atomically — e.g. "migrate this instance to build v2.1.4 AND move the token to the Approval node AND set variable approved = true."
Composable Action Panel
Move Token → pick target node
Migrate Instance to Build → pick target build
Change Variables → edit JSON patch
  • Accessible from both single-instance detail and bulk list selection
  • Design as a step-by-step wizard or composable action panel
  • Any combination of the three is valid (all three, any two, or just one)
  • Operations execute in logical order: Change Variables → Migrate Instance → Move Token
  • Summary/review step before executing: what will change and on how many instances
  • Show per-instance result after execution (success / partial failure / failed)

Entry Points

① Via Process Instances

Operator comes from the Process Instances monitoring list, selects one or more instances (filterable by process definition, status, build version, date range, initiator), and triggers corrective actions from a toolbar or row context menu.

② Via UI Flow Sessions

Operator opens a UI Flow Session, sees all process instances grouped under it, and applies corrective actions on individual instances or all instances in the session at once (session-level bulk).

UX Principles

🔒

Safe by Default

Every action requires a confirmation step with a clear impact summary: X instances affected, current state → new state.

📋

Auditable

Read-only audit trail per instance: who performed which corrective action and when.

🔍

Filterable at Scale

Filter by process name, status, build version, date range, initiator. Operators work with large volumes.

👥

Bulk-First Thinking

Primary workflow assumes many instances at once (e.g. after a bad deployment), not just one.

⏳

Async Feedback

Operations may take time. Show per-instance progress and result: in-progress, success, partial failure.

🎨

Status-Aware Styling

STARTED = yellow · FAILED/ERROR = red · EXPIRED/ABORTED/TERMINATED = orange · FINISHED = green · ON_HOLD = grey

Not in Scope

  • The BPMN process editor
  • Build deployment or release flows
  • Authentication or permission management screens

Deliverable

A complete interactive prototype of the Corrective Actions screen covering all four features, both entry paths (Process Instances and UI Flow Sessions), single and bulk operation flows, and the combined action wizard. Use a dark-neutral enterprise design language consistent with a monitoring/operations console.
